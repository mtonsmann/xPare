# Release Model

xPare follows the normal open-source macOS pattern:

- **Source is the canonical distribution** for contributors and privacy-sensitive
  users (`make app` / `make run` build and launch a local bundle).
- **Tagged GitHub Releases** (`v*`) are the end-user download surface. The asset is
  a **Developer ID signed, notarized, stapled `.app` in a zip**. Releases are
  created as **drafts** ‚Äî a human reviews and publishes; nothing ships on tag push
  alone. A DMG is deferred (see [`deferred-work.md`](deferred-work.md)).
- **Releases are arm64-only (Apple Silicon) at 1.0.** x86_64 and universal binaries
  are deferred until there is demand (see [`deferred-work.md`](deferred-work.md)).
- The current version is **1.0.0-rc.1**, heading to 1.0.0.

Unlike the upstream FormatStripper lineage, the xPare Swift app **statically
links** `libxpare_ffi.a`, so a release bundle has **no embedded dylib** under
`Contents/Frameworks` to relocate or sign separately ‚Äî the bundle is the Swift
executable plus its `Info.plist` and icon, and the Developer ID signature covers the
single Mach-O and the bundle.

## Building locally

From the repo root:

```sh
make app     # build + assemble shells/macos/dist/xPare.app (ad-hoc signed)
make run     # the above, then `open` it
```

For an explicitly **unsigned preview** archive (no Apple account required):

```sh
make preview                 # uses an exact vX.Y.Z tag, or:
make preview VERSION=1.2.3
```

That writes `dist/release/xPare-vX.Y.Z-macos-<arch>-unsigned-preview.zip` and
its `.sha256`. Preview archives are test artifacts, not official binaries: they are
unsigned/ad-hoc and must not be promoted as end-user downloads.

## Release channels

1. GitHub Releases for users (notarized zip, arm64-only at 1.0).
2. Build-from-source for contributors and privacy-sensitive users.
3. Homebrew (personal tap, then a `homebrew/cask` submission) is deferred ‚Äî it
   requires published notarized releases first; see
   [`deferred-work.md`](deferred-work.md).

Signed assets are architecture-labeled:

```text
xPare-vX.Y.Z-macos-arm64.zip
```

If the project later ships x86_64 or universal binaries, asset names must keep the
architecture choice explicit.

## Release assets

A published release carries, alongside the notes:

- `xPare-vX.Y.Z-macos-arm64.zip` ‚Äî the notarized, stapled app bundle.
- `xPare-vX.Y.Z-macos-arm64.zip.sha256` ‚Äî per-asset checksum
  (basename-relative, so `shasum -c` works from the download directory).
- `SHA256SUMS` ‚Äî an aggregate checksum file written by `dist` into
  `dist/release/` and uploaded by `github-release` alongside the zip.
- A **build-provenance attestation** (`actions/attest-build-provenance`) over the
  zip and `SHA256SUMS`, verifiable with `gh attestation verify`.
- An **SPDX-JSON SBOM** (`xPare-vX.Y.Z-sbom.spdx.json`, generated by syft over the
  workspace) attached to the release.

## Release workflow

`.github/workflows/release.yml` runs on `v*` tags. The tag version must match
`^[0-9]+(\.[0-9]+){2}(-[0-9A-Za-z]+(\.[0-9A-Za-z]+)*)?$` after the leading `v` ‚Äî
i.e. strict three-component semver with an optional dotted prerelease, so
`v1.0.0-rc.1` is a valid tag. A fail-closed step also asserts the tag version
equals the workspace version in `Cargo.toml`; a mismatch fails the release.

1. **validate** (always): checkout ‚Üí require a successful Release Fuzz run on the
   exact tag SHA with a sufficient per-target budget (see the fuzz gate below) ‚Üí
   `cargo run --locked -p xtask -- ci` (the full gate) ‚Üí build the FFI staticlib +
   `swift build -c release` ‚Üí `make preview` ‚Üí upload the unsigned preview as a
   workflow artifact.
2. **publish-official** (gated): only when repo variable
   `XPARE_ENABLE_OFFICIAL_RELEASE == 'true'` **and** the Apple secrets are
   present (it must also export `NOTARY_PROFILE` ‚Äî `dist` refuses to run without
   it). It re-checks `shells/macos/xPare.entitlements`, imports the Developer ID
   cert into a temp keychain, stores notary credentials, runs `make dist` and
   `make github-release` (the fail-closed signing/notarization flow below),
   attaches the provenance attestation and SBOM, then wipes the signing material.

Releases are created with `--draft` (and `--prerelease` for any hyphenated
version such as `1.0.0-rc.1`); a maintainer publishes the draft manually after
inspecting it. Re-running `github-release` is idempotent: assets are re-uploaded
with `gh release upload --clobber` instead of failing.

Signing/notary credentials are **never** required for pull-request CI. Absent them,
CI still builds and tests, but must not publish an artifact as an official binary.

## Release fuzz gate

Run the manual Release Fuzz workflow on the release-candidate ref before tagging
or rerunning the final release:

```sh
gh workflow run release-fuzz.yml --ref v1.0.0-rc.1 -f minutes_per_target=10
```

The workflow uses the same `cargo xtask check-fuzz` path as local `make
fuzz-smoke`, but with a release-scale per-target budget and uploaded corpus/crash
artifacts. The release workflow queries GitHub Actions for a successful Release
Fuzz run whose `head_sha` is the exact release tag commit **and** whose per-target
budget meets the policy:

- **Prerelease tags** (any version containing `-`, e.g. `v1.0.0-rc.1`):
  **‚â• 10 minutes per target**.
- **Final releases**: **‚â• 30 minutes per target**.

The budget is parsed from the run's display title
(`Release fuzz (<ref>, <N>m/target)` ‚Äî that run-name format is load-bearing and
commented as such in both workflows). If the final tag points at a different SHA,
or the run's budget was below policy, the release fails until fuzz is rerun.

## The fail-closed signing and notarization flow

`make dist` (which drives `shells/macos/release.sh dist`) is fail-closed at every
step; `cargo xtask check-release-posture` mechanically asserts the guards stay
wired:

- **Refuses to run without `NOTARY_PROFILE`** (`xcrun notarytool
  store-credentials`). Official assets must be notarized ‚Äî use `preview` for
  unsigned archives.
- Builds with `cargo build --locked`, signs the executable and bundle with the
  checked-in `shells/macos/xPare.entitlements` (alternate resolved entitlement
  paths are rejected), and verifies both signed payloads with
  `codesign -d --entitlements - --xml` ‚Äî the signed payload must contain only
  `com.apple.security.app-sandbox = true`. A separate check verifies the
  Hardened Runtime flag in the CodeDirectory.
- Submits to notarization with `notarytool --output-format json` and **parses the
  verdict**: anything other than `Accepted` fails, printing the
  `xcrun notarytool log <id>` command for diagnosis. The temporary submission zip
  is deleted after stapling.
- `spctl --assess` (Gatekeeper) is **fatal** on failure, not advisory.
- `github-release` unpacks the final zip and runs `stapler validate` on the
  contained app **before** uploading ‚Äî an unstapled app never becomes a release
  asset.

### Local official release (with credentials)

On an exact `vX.Y.Z` tag, with full Xcode + a Developer ID identity:

```sh
xcrun notarytool store-credentials xpare-notary \
  --apple-id you@example.com --team-id TEAMID --password app-specific-password

make dist VERSION=1.0.0 \
  CERT_NAME="Developer ID Application: Your Name (TEAMID)" \
  NOTARY_PROFILE=xpare-notary

make github-release VERSION=1.0.0   # creates a DRAFT release
```

## Versioning and stability

From 1.0, xPare follows **semver**, and the public surface the version protects
is:

- the **C ABI** (`core-ffi/include/xpare.h` ‚Äî frozen at ABI v3; see the
  [FFI guardrail](guardrails/ffi-boundary-and-abi-stability.md)),
- the **config schema** (`CONFIG_VERSION`; deliberately strict ‚Äî unknown
  fields/versions are rejected fail-closed, and evolution happens only via
  explicit `CONFIG_VERSION` bumps),
- the **CLI flags**.

A breaking change to any of these requires a **major** version.

**MSRV policy:** the minimum supported Rust version is declared as `rust-version`
in the root `Cargo.toml` and checked by a dedicated CI job (a floor-compile of the
workspace on that exact toolchain). An MSRV bump is a **minor** release, called
out in the release notes.

**Deprecation policy:** a deprecated config field or CLI flag keeps working for at
least one minor release cycle while emitting a warning; removal happens only at a
major version.

## Updates

xPare has **no auto-update mechanism, by design**: an update checker would be a
network call, and the no-network posture (no network code anywhere, no network
entitlement) is the product's core privacy guarantee ‚Äî see
[`SECURITY.md`](../SECURITY.md). Updates are **manual**: download the new release
from GitHub Releases (or pull and rebuild from source). The app shows its version
in the menu (`xPare vX.Y.Z`) so users can compare against the latest release.

## First public binary prerequisites

- **Confirm the public bundle identifier before the first notarization**
  (currently `com.xpare.app`). Notarization history and Gatekeeper identity attach
  to the identifier ‚Äî treat it as frozen once the first official build is
  notarized. **Must be confirmed before the first notarization run.**
- Add Developer ID signing + notarization secrets (kept **out** of the repo).
- Keep official signing tied to `shells/macos/xPare.entitlements`; a signed
  app without the minimal App Sandbox-only payload is not an official xPare
  release.
- Verify Gatekeeper acceptance on a clean macOS machine.
- Keep the no-network, no-telemetry, no-clipboard-content-logging posture intact ‚Äî
  enforced by `cargo xtask ci` on every release run.
